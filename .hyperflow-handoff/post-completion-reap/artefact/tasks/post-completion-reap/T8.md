# T8 — Config docs + CHANGELOG + end-to-end verification

**Specialist:** `docs`, `devops-reviewer` · **Batch 4** · depends T6 (and ideally after T7)

## Task
Document the `cleanup` config, add a CHANGELOG entry, and run an end-to-end verification proving the whole reap subsystem behaves per the disposition policy — archive, preserve, delete, report, idempotency, dry-run.

## Why
The `cleanup` keys were previously undocumented in the config reference and unschema'd; the feature needs a user-facing doc + a release note, and a real end-to-end run is the acceptance proof for the whole task file.

## Scope
- IN: `docs/orchestration.md`, `CHANGELOG.md`, and a verification run (no new product code).
- OUT: no engine/skill changes (fix-forward via the owning sub-task if verification fails).

## Files in scope
- `docs/orchestration.md` — document the full `cleanup` block (`auto`, `staleDays`, `pruneDays`, `reapOnComplete`, `usageRetentionDays`, `logMaxLines`, `dryRun`) and the reap lifecycle; extend the existing cleanup mention (`:326-334`).
- `CHANGELOG.md` — add an entry: post-completion reap (auto at dispatch/deploy/handoff + `/hyperflow:reap`), archive-first, memory-preserving, unbounded-log/usage retention, config now schema-validated.

## Verification (the end-to-end acceptance for the whole feature)
Build a temp `.hyperflow/` fixture containing: a completed flat task + its brief dir + a viewer JSON twin + one durable memory learning + a stale `usage/*.jsonl` + a bloated `.session-start.log` + a `pi-first-class-support.draft.md`-style orphan. Then:
1. `reap.py --slug <fixture> --dry-run` → report lists the plan; **filesystem unchanged**.
2. `reap.py --slug <fixture>` → task/brief-dir/spec/twin moved to `archive/`; durable learning still present; stale usage deleted; log truncated to `logMaxLines`; report accurate.
3. `reap.py --slug <fixture>` again → clean no-op.
4. Traversal slug → refused.
5. `cleanup.reapOnComplete=false` → the lifecycle hooks (T6) do not fire.

## Acceptance criteria
1. `cleanup.*` fully documented with defaults + effect.
2. CHANGELOG entry present.
3. All five verification steps pass and are recorded in the PR/commit body.
4. `npm test` / the Python test suite (`test_config_cleanup`, `test_archive_artefacts`, `test_reap`) green.

## Test cases
- The five verification steps above (each an assertion).
- **E2E:** the full fixture reap sequence is the E2E — capture the report output as evidence.

## Gotchas
- Verify on a COPY/fixture — never reap the repo's real in-flight tasks (`full-codex-support`, `visual-artefacts`) during testing.
- If a verification step fails, route the fix to the owning sub-task (T2/T3/T6), don't patch here.

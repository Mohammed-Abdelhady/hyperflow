# T1 — `cleanup` config block (schema + defaults)

**Specialist:** `security-reviewer`, `devops-reviewer` · **Batch 1** · parallel with T2

## Task
Make the `cleanup` config a first-class, schema-validated block. Today `scripts/archive-artefacts.py` already reads `cleanup.{auto,staleDays,pruneDays}` from `~/.hyperflow/config.json`, but `config/schema.json` has no `cleanup` property and its root is `additionalProperties:false` — so any user who sets a `cleanup` block **fails validation**. This task legalizes the existing keys and adds the reap knobs.

## Why
The reaper (T3) and the lifecycle wiring (T6) gate on `cleanup.reapOnComplete` and read `usageRetentionDays`/`logMaxLines`. Those must be valid config, not ad-hoc script defaults.

## Scope
- IN: add a `cleanup` object to schema + (opinionated) defaults; a focused schema test.
- OUT: no consumer changes (T2/T3 read it); no changes to other config blocks.

## Files in scope
- `config/schema.json` — add a `cleanup` object under `properties`, mirroring the `memory` block shape (see `schema.json:105-117`). Properties, all with `default` + bounds, `additionalProperties:false`:
  - `auto` (boolean, default `true`) — master switch for the daily sweep.
  - `staleDays` (integer, default `7`, minimum `1`) — existing archiver knob.
  - `pruneDays` (integer, default `30`, minimum `1`) — existing archiver knob.
  - `reapOnComplete` (boolean, default `true`) — fire reap at lifecycle termini.
  - `usageRetentionDays` (integer, default `30`, minimum `1`) — `usage/*.jsonl` retention.
  - `logMaxLines` (integer, default `2000`, minimum `100`) — `.session-start.log` cap.
  - `dryRun` (boolean, default `false`) — global reap dry-run override.
- `config/defaults.json` — follow the existing convention: defaults.json carries only *opinionated non-default* values (memory/context are NOT in it). Add `cleanup` **only** if a value should differ from the schema default; otherwise leave it out and rely on the schema `default`s. Document the choice in the PR body.
- `tests/test_config_cleanup.py` — new.

## Acceptance criteria
1. A config with a full `cleanup` block validates against `schema.json`.
2. An unknown key inside `cleanup` is rejected (`additionalProperties:false`).
3. Out-of-range values (`logMaxLines: 10`) are rejected by the `minimum`.
4. Existing top-level blocks still validate unchanged.

## Test cases (real input → outcome)
- `{"cleanup":{"auto":true,"staleDays":7,"pruneDays":30,"reapOnComplete":true,"usageRetentionDays":30,"logMaxLines":2000,"dryRun":false}}` → valid.
- `{"cleanup":{"reapOnComplete":true,"bogus":1}}` → invalid (additionalProperties).
- `{"cleanup":{"logMaxLines":10}}` → invalid (minimum 100).
- `{}` (no cleanup) → valid (all defaults apply).
- **E2E:** run the repo's existing schema-validation test/CLI against a sample `~/.hyperflow/config.json` that sets `cleanup` → passes where it previously errored.

## Related context
- Schema block pattern: `config/schema.json:105-117` (`memory`), root `additionalProperties:false` at `:267`.
- Existing reader + defaults: `scripts/archive-artefacts.py:25-29` (`DEFAULTS`), `:45`, `:63-75` (`load_cfg`).
- Docs mention the keys: `docs/orchestration.md:326-334`.

## Gotchas
- Do NOT remove `additionalProperties:false` from the root — add the nested object instead.
- Keep defaults byte-consistent with `archive-artefacts.py` `DEFAULTS` so schema and script agree.

# T4 — `skills/reap/SKILL.md` (command + reap-phase contract + report render)

**Specialist:** `architect`, `devops-reviewer` · **Batch 3** · depends T3

## Task
Author the `reap` skill: the standalone `/hyperflow:reap <slug>` command AND the reusable "reap phase" contract that dispatch/deploy/handoff (T6) invoke. It wraps `scripts/reap.py`, renders the Reap Report block, and appends the ledger.

## Why
The user wants reap runnable by hand and to "show what he cleaned." The skill is the human-facing surface over the engine; the phase contract is what the lifecycle skills call so behavior is identical auto or manual.

## Scope
- IN: `skills/reap/SKILL.md` only.
- OUT: no manifest registration (T5); no lifecycle edits (T6); no engine changes (T3).

## Contents the SKILL.md must define
- **Frontmatter** matching the repo's other skills (name, description, trigger phrases: `/hyperflow:reap`, "clean up the finished task", "reap", "garbage collect this task").
- **`/hyperflow:reap <slug> [--dry-run] [--force]`** — resolves the slug, calls `reap.py`, renders the report. Default is a **dry-run preview then a confirm** when invoked manually on a slug that is NOT in terminal state (safety); a terminal slug reaps directly. `--force` skips the terminal-state check with a printed warning.
- **Reap-phase contract** (the reusable block): the exact call the lifecycle skills make — `reap.py <root> --slug <slug>` gated on `cleanup.reapOnComplete`, capture JSON, render block, append `archive/.reap-log.jsonl`. Document that it is a no-op when `reapOnComplete=false` or `dryRun=true`.
- **Reap Report block** format (the "show what was cleaned" surface):
  ```
  ── Reap: <slug> ──
  Archived   : <n> → .hyperflow/archive/… (task, brief dir, spec, twin)
  Deleted    : <n> ephemeral (usage ×k, log −Xk lines, bg ×m)
  Memory     : index rebuilt · <o> orphaned refs dropped · <c> compacted
  Freed      : <bytes>   Mode: live|dry-run
  ```
- **Never touches source code** — reap is cache-scoped only. State this explicitly (mirrors `cache`/`background`/`flush` skill guardrails).

## Acceptance criteria
1. `/hyperflow:reap <slug>` on a completed slug reaps and prints the report.
2. On a non-terminal slug, manual invocation previews (dry-run) and asks to confirm; `--force` overrides.
3. The reap-phase contract block is copy-usable by T6 verbatim.
4. Report block renders all four lines from the engine's JSON.
5. Skill declares it never edits source.

## Test cases
- Manual reap of a completed fixture slug → report shown, archive populated.
- Manual reap of an in-progress slug → dry-run preview + confirm gate.
- `--dry-run` → report marked dry-run, nothing changes.
- **E2E:** invoke the skill against the T3 fixture; confirm the rendered block matches the engine JSON.

- Keep the SKILL.md within the repo's per-file line budget; factor detail into the phase-contract block rather than prose.

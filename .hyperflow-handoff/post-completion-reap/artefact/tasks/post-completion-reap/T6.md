# T6 — Wire lifecycle hook-ins (dispatch / deploy / handoff)

**Specialist:** `backend-reviewer`, `security-reviewer`, `architect` · **Batch 3** · depends T4

## Task
Invoke the reap phase (T4 contract) at each true lifecycle terminus, gated on `cleanup.reapOnComplete`, and replace dispatch's ad-hoc task-file delete with the reap phase so there is one disposition path.

## Why
The three termini are where a task is "finished." Today dispatch hard-deletes just the task file (`:324`), deploy cleans nothing, and handoff only archives its own package. Wiring reap here delivers the user's "when he finishes, go clean this task — the whole scope — then show what he cleaned."

## Scope
- IN: prose/step edits to three lifecycle skills to call the reap phase + print the report.
- OUT: no engine/skill-content changes (T3/T4); no doctrine (T7).

## Files in scope
- `skills/dispatch/SKILL.md` — **Step 4 Wrap Up** (`:318-335`): replace the ad-hoc "delete the completed task file" (`:324`) with an invocation of the reap phase for the finished `<slug>` (flat delete becomes archive-first reap of the whole scope). Keep the memory `chore(memory):` commit ordering. Print the Reap Report. Gate on `cleanup.reapOnComplete`; skip on a handoff build that will deploy (let deploy reap).
- `skills/deploy/SKILL.md` — after **Step 7 Output** (`:194-205`): add a terminal reap-phase invocation for the shipped slug (deploy currently has no GC at all). Gate on `cleanup.reapOnComplete`; print the report under the Ship Result block.
- `skills/handoff/SKILL.md` — **`complete` subcommand** (`:58-60`): after archiving the handoff package, invoke the reap phase for the slug so the task file / brief dir / twin / memory-optimize also happen (today only the package moves). Print the report.

## Acceptance criteria
1. dispatch Step 4 reaps the whole finished scope (not just the task file), archive-first, and prints the report; ordering with the memory commit preserved.
2. deploy reaps the shipped slug after a successful push and prints the report.
3. handoff `complete` reaps the slug in addition to archiving the package.
4. All three are no-ops when `cleanup.reapOnComplete=false`.
5. No terminus reaps a non-terminal slug (dispatch that stops at its Step-5 gate without completion does not reap).

## Test cases

- handoff `complete <slug>` → package archived AND slug scope reaped.
- **E2E:** run a mini dispatch→deploy on a fixture; confirm reap fires once at the correct terminus and the report appears.

## Gotchas
- Avoid double-reap on the deploy path: if dispatch already reaped and deploy runs later on the same slug, the second reap is a harmless no-op (idempotent) — but prefer dispatch to SKIP reap when it knows deploy will follow (handoff `on_complete=deploy`).
- Handoff `complete` must still leave `.hyperflow-handoff/.archive/` handling to handoff itself; reap only touches the `.hyperflow/` scope, never the handoff package.
- Preserve each skill's existing gate wording (audit/deploy gates stay explicit).

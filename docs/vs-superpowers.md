# Hyperflow vs Superpowers (honest)

Different jobs. Use both ideas; do not pretend one repo replaces the other.

| | Superpowers | Hyperflow |
|---|---|---|
| Primary job | General SDLC methodology for coding agents | Project OS: memory + doctrine + reviewed chain |
| Strength | Spec discipline, TDD culture, huge install base | Multi-session memory, specialists, monorepo/pro gates, handoff |
| Weakness for pros | Easy to stay generic across repos | Heavier; needs progressive disclosure |
| Best default | Greenfield / skill teaching | Ongoing product repos (Uptend, TIP, internal platforms) |

## What we steal (habits, not dependency)

- Junior-proof plans ([templates/plan-habits/junior-proof-tdd.md](../templates/plan-habits/junior-proof-tdd.md))
- Chunked design sign-off
- YAGNI in decomposition

## What we will not copy

- Marketplace star chasing as a north star
- Dropping project memory to look simpler

## Positioning line

**Superpowers teaches agents how to build. Hyperflow teaches them how to stay correct on *your* repo across sessions.**
